Техническое задание на разработку платформы AttackChainGenПроект: Синтетический генератор цепочек атак и фоновых событий (Web-платформа)Исполнитель / Команда: Антигравити1. Введение и цели продукта1.1. Назначение системы:Разработка полноценного веб-приложения (AttackChainGen) для генерации синтетических событий информационной безопасности (ИБ) и их отправки в Elastic Security. Система предназначена для обучения аналитиков SOC, тренировки навыков Threat Hunting и тестирования правил детектирования (SIEM Rules).1.2. Основные цели:Генерация связных цепочек событий (Kill Chain), где артефакты (PID, IP, пользователи) логически наследуются от события к событию.Параллельная генерация фонового "шума" (легитимной активности) для усложнения задач аналитиков.Предоставление удобного веб-интерфейса для преподавателей/инструкторов для визуального создания сценариев и управления запусками.Полная совместимость генерируемых данных с Elastic Common Schema (ECS 8.x).2. Архитектура системыПлатформа представляет собой клиент-серверное веб-приложение с асинхронным ядром генерации.2.1. Web Stack (Уровень управления)Frontend: React или Vue.js (SPA для панели управления, визуального редактора сценариев и мониторинга).Backend API: Python + FastAPI (REST API для фронтенда, валидация данных через Pydantic).Database: PostgreSQL (хранение пользователей, настроек стендов, библиотеки сценариев и истории запусков).Task Broker: Redis + Celery (критически важный компонент для управления длительными асинхронными задачами генерации).2.2. Generator Engine (Уровень исполнения)Логика генератора работает внутри Celery Workers и состоит из модулей:Playbook Parser: Читает и валидирует YAML/JSON сценарии атак.Context Manager (Менеджер состояния): Ядро генератора. Хранит глобальные переменные инцидента и динамические артефакты (например, сгенерированный process.pid), передавая их дочерним событиям в цепочке.Template Engine: Шаблонизатор (Jinja2/python-box) с эталонными шаблонами логов (Sysmon, Windows Security).Noise Generator: Независимый микро-поток, генерирующий легитимные события.Elastic Exporter: Транспортный модуль (на базе elasticsearch-py) для отправки готовых JSON-документов через Bulk API в Data Streams.3. Ролевая модель и администрированиеСистема поддерживает две основные роли:3.1. Роль: Администратор (Руководитель SOC / Главный преподаватель)Управление пользователями (создание, удаление, сброс паролей).Настройка подключений к целевым SIEM (URL Elasticsearch, API ключи, индексы). Поддержка множества стендов (например, "Стенд Группы А", "Стенд Группы Б").Управление глобальными словарями для Faker (загрузка реалистичных внутренних IP-сетей, доменных имен корпорации).Управление библиотекой шаблонов (добавление новых типов сырых логов JSON с Jinja2-переменными).3.2. Роль: Инструктор (Пользователь)Создание и редактирование сценариев атак (Playbooks).Запуск симуляций на доступных стендах.Просмотр истории запусков и артефактов (шпаргалок для проверки студентов).4. Пользовательский интерфейс и функциональные требования4.1. Визуальный редактор сценариев (Playbook Builder)Интерфейс должен позволять создавать сложные YAML-сценарии без ручного написания кода (Node-based редактор или карточки шагов).Global Context: Поля для задания целевого хоста, пользователя или выбора их случайной генерации.Steps (Шаги): Добавление шага из библиотеки шаблонов (например, Process Creation, Network Connection).Связывание (State Binding): В полях ввода (например, process.parent.pid) пользователь должен иметь возможность выбрать из выпадающего списка "Взять PID из Шага 1".Timeline: Настройка временных задержек между шагами (секунды, минуты, часы).Code View: Вкладка для просмотра/редактирования итогового YAML-кода сценария.4.2. Механизм Time Drift (Сдвиг времени)Система должна поддерживать два режима генерации:Real-time: События генерируются в реальном времени с заданными задержками (Worker засыпает между шагами).Historical (Backdating): Генерация событий "в прошлом" (например, 3 дня назад). Worker вычисляет таймстемпы и генерирует логи без пауз, что позволяет сразу приступить к расследованию.5. Процесс запуска симуляции (Execution Flow)Процесс разделен на 4 этапа, чтобы обеспечить неблокирующую работу веб-интерфейса:Этап подготовки (Frontend):Инструктор выбирает сценарий, целевой стенд, задает переопределение переменных (например, user.name="admin"), выбирает режим времени (Real-time/Historical) и уровень шума. Нажимает "Run".Этап маршрутизации (FastAPI):Backend принимает POST-запрос, валидирует данные, создает запись в PostgreSQL (Status: Pending) и отправляет задачу в брокер (Redis). Frontend моментально получает ответ {status: "accepted", run_id: 1042}.Этап выполнения (Celery Worker):Worker берет задачу, инициализирует Context Manager.Запускает фоновый Noise Generator.Итерируется по шагам сценария (Execution Loop): берет шаг, генерирует артефакты (сохраняя их в контекст), шаблонизирует JSON, отправляет в Elastic, "засыпает" до следующего шага (в Real-time режиме).По завершении останавливает шум.Этап обратной связи (Web Monitoring):Celery обновляет статус прогресса (например, "4/10 шагов"). Frontend получает эти данные (через Polling или WebSockets) и отображает прогресс-бар. После завершения доступна кнопка "View Artifacts" для просмотра сгенерированных IP/PID злоумышленника.6. Формат сценариев (Playbooks) под капотомВне зависимости от UI, ядро оперирует YAML-сценариями. Пример структуры:name: "Spearphishing Attachment leading to C2"
mitre_tactics: ["TA0001", "TA0002", "TA0011"]
global_context:
  host.name: "DESKTOP-HR-01"
  user.name: "j.doe"

steps:
  - id: "step_1_initial_access"
    delay_from_start: "0s"
    template: "sysmon_event_1"
    fields:
      process.name: "WINWORD.EXE"
      process.command_line: "C:\\Program Files\\...\\WINWORD.EXE"
    
  - id: "step_2_execution"
    depends_on: "step_1_initial_access" # Наследование артефактов (parent.pid)
    delay_from_prev: "15s"
    template: "sysmon_event_1"
    fields:
      process.name: "powershell.exe"
      process.command_line: "powershell.exe -nop -w hidden -c IEX..."
